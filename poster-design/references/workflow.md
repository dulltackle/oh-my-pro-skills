# 强制工作流

> **⚠️ 首版设计必须按此流程执行。** 但可根据风险等级选择不同路径。
> 修改迭代（v2+）可简化，但 Phase 5 审查不可跳过。

## 风险分级

拿到需求后先判断风险等级，选择对应路径。**默认为 🔴 高风险，只有确认满足低风险条件时才可降级。**

| 等级 | 条件 | 必走阶段 | 可简化 |
|------|------|---------|--------|
| 🟢 低 | 信息 ≤5 条 + 无照片 + 用户无高质量要求 | Phase 2(快速)→4→5 | Phase 1 精简版，跳过 3 |
| 🟡 中 | 有照片 或 信息 5-10 条 | Phase 1(精简)→2→3→4→5 | Phase 1 跳过 1.4 动线预演 |
| 🔴 高 | 信息 10+ 条 或 用户要求高 或 多张照片 | Phase 1-5 全部 | 无 |

Phase 5 在所有级别都不可跳过。

### 精简版 Phase 1（🟢低 / 🟡中风险）

低/中风险时，Phase 1 简化为：
1. 快速列出核心信息（🟢 可不必须做三级分类）
2. 用 `scripts/space_budget.py --layout X` 快速算字号
3. 有图片时做素材检查（见 `image-checklist.md`），无图跳过

## 流程总览

```
Phase 1: 预设计决策 → Phase 2: 设计方案锁定 → Phase 3: 排版细节
    → Phase 4: 写代码出图 → [🔒 门控: 视觉检查] → 发送用户
                                                    ↓ 不通过
                                              修复 → 重新截图 → [🔒 门控]
```

---

## Phase 1：预设计决策（首版必做）

> 详细清单见 `pre-design-checklist.md`
> 涉及人物照片时同时读 `image-checklist.md`

### 1.1 信息审计
- 列出所有信息项 → 🔴🟡⚪ 三级分类 → 砍掉 ⚪ 级冗余
- 硬规则：⚪ 超过 5 个必须砍或合

### 1.2 空间预算
- 各区域占比分配（照片≤45%、标题≤18%、每内容区≥10%）
- 从区块高度反推字号（先定正文≥28px，再反推标题）
- 可配合 `scripts/space_budget.py --layout A` 辅助计算

### 1.3 素材检查（写代码前的硬门槛）

收到图片后、写代码前，必须按 `image-checklist.md` 完整执行。不通过不写代码。

### 1.4 动线预演
- 3 秒浏览路径模拟：第1眼→第2眼→第3眼→CTA
- 确认无断裂、无多元素抢焦点

---

## Phase 2：设计方案锁定

按顺序读 3 个文件，每个做**一个决策**并记录：

| 顺序 | 读什么 | 决策什么 |
|------|--------|---------|
| 2a | `design-directions.md` | 选 1 个主方向（可融合 2 个） |
| 2b | `layout-patterns.md` | 根据内容量+有无照片选布局模式 |
| 2c | `color-strategy.md` | 从方向对应的调色板选色，定义 CSS 变量 |

完成后应能回答："这张海报用什么方向、什么布局、什么色板"。

---

## Phase 3：排版与细节

| 顺序 | 读什么 | 关注点 |
|------|--------|--------|
| 3a | `typography.md` | 字体配对、尺度跳跃（Hero:Section:Body ≥ 2.5:1）、中文排版 |
| 3b | `image-cropping.md` | 根据 Phase 1.3 的素材检查结果选裁剪方式和 CSS 实现 |
| 3c | `poster-anatomy.md` | 各区块角色确认 |

反模式快速扫 `anti-patterns.md` 的 **B 类（实现阶段）** 条目。

---

## Phase 4：写代码 + 出图

1. 用 `scripts/scaffold.py --layout X` 生成 HTML 骨架，或按 `tailwind-recipes.md` 组件指南拼装
2. Puppeteer 截图（1080×1920 @2x）

---

## Phase 5：🔒 双轨出图审查（每一版都必须执行）

> **详细规范见 `visual-review.md`。这是发出图片给用户的硬门槛。**

核心要点：
1. **轨道 A（DOM 层）**：检查文字溢出、图片加载、元素溢出
2. **轨道 B（视觉层）**：读取 PNG 用视觉能力按 5 维度（完整性、比例空间、层级焦点、协调质感、细节精度）评分
3. **判定**：🟢 放行 / 🟡 小修后重审 / 🔴 返工回对应 Phase

### ❌ 绝对禁止

- 截图后不查看就直接发送给用户
- 只看 DOM 检查结果就跳过视觉层检查
- 用户指出问题后才补审查（应该在用户看到之前就拦截）

### 经验沉淀

用户主动触发（输入"请沉淀本次生成的经验和教训"），流程见 `experience-bank.md`。

## 修改迭代（v2+）的简化流程

当用户要求修改已有海报时：

```
理解修改意图 → 定位涉及元素 → 读对应 reference → 改代码
    → 截图 → 🔒 Phase 5 审查（完整 5 维度）→ 发送
```

注意：
- 即使只改一个字号，也要截图 + 视觉检查确认没有破坏其他元素
- 连续修改 ≥ 3 次仍未满足用户时，建议回到 Phase 1 重新梳理

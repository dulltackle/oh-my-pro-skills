# 交付物约定

本文件用于说明 `brainstorming-pro` 何时写 spec、何时写实施计划，以及文档里至少要包含什么。

## spec

### 什么时候写

- 用户明确要求先出 spec
- 设计风险较高，需要书面化关键决策
- 后续实现会由低上下文执行者接手

### 建议写什么

- 目标、范围、成功标准
- 明确不做什么
- 关键结构或模块划分
- 接口边界、数据流或状态流
- 错误处理、异常路径、边界条件
- 验证思路或验收方式

### 默认路径

- 如果仓库已有既定文档约定，优先遵循现有约定
- 如果没有明确约定，默认写到 `docs/specs/YYYY-MM-DD-<topic>-design.md`

## 实施计划

### 什么时候写

- 用户明确要求先出实施计划
- spec 已确认，需要把实现拆成低上下文可执行步骤
- 任务较大，直接实现容易遗漏路径、验证或拆分边界

### 建议写什么

- 目标和总体实现方式
- 任务拆分顺序
- 每步涉及的文件路径
- 每步的验证或测试命令
- 预期结果
- 必要时的伪代码或最小代码骨架

### 粒度要求

- 单步应尽量控制在约 2-5 分钟可完成
- 删除投机性任务，保持 DRY 和 YAGNI
- 不要让某一步同时承载多个职责

### 默认路径

- 默认写到 `docs/plans/YYYY-MM-DD-<topic>-implementation-plan.md`

### 计划头部模板

```md
# [功能名] 实施计划

**目标：** [一句话说明目标]

**架构：** [2-3 句说明总体实现方式]

**技术栈：** [关键技术或依赖]

---
```

## 审阅入口

### spec 审阅

- 写完 spec 后先做一次文档审阅
- 若环境允许委派，可按需使用独立审阅 agent
- prompt 模板见 `references/spec-document-reviewer-prompt.md`

### 实施计划审阅

- 写完实施计划后同样先做审阅
- prompt 模板见 `references/plan-document-reviewer-prompt.md`
- 审阅重点是：是否覆盖 spec、任务是否原子、路径与验证是否明确、是否存在占位符或范围漂移

## 用户确认门禁

- spec 审阅通过后，先请用户确认 spec
- 实施计划审阅通过后，再请用户确认实施计划
- 只有本次任务要求的文档都已确认后，才允许进入实现

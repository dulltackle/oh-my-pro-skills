# 设计推进方式

本文件用于说明 `brainstorming-pro` 在不同任务下如何推进，但它不是要求每次都完整执行的固定流水线。

## 先分流，再推进

默认先判断当前请求属于哪一类：

### 1. 设计对齐

- 目标：把模糊需求收敛成明确方向
- 常见输入：一句想法、半成品需求、存在多种做法的设计题
- 常见输出：设计结论、方案比较、关键取舍、下一步建议

### 2. 文档收敛

- 目标：把已确认方向写成 spec 或实施计划
- 常见输入：用户已经认可总体方案，或明确要求先出文档
- 常见输出：可审阅 spec、可执行实施计划、审阅结论

### 3. 大任务拆分

- 目标：把过大的请求拆成几个可独立理解和交付的子问题
- 常见输入：同时覆盖产品端、后台、权限、计费、分析等多块内容
- 常见输出：子项目边界、依赖关系、建议顺序

## 逐轮澄清，而不是一次性盘问

- 每次只问一个问题
- 优先问目标、约束、边界、成功标准
- 如果上下文已经给出答案，不要重复追问
- 能用选项题时优先用选项题，但不要为了形式牺牲表达准确性

## 方案比较的最低要求

默认给出 2-3 种可行方案，并说明：

- 各自适用条件
- 收益和风险
- 复杂度或维护成本
- 为什么推荐其中一个

如果上下文已经强烈收敛到单一路径，可以不强凑多个方案，但要明确说明为什么现在不需要比较。

## 什么时候升级到文档

满足任一情况时，通常应该继续产出 spec 或实施计划：

- 用户明确要求先出 spec、plan、设计稿说明或实施拆解
- 任务较大，单靠对话结论难以保证后续实现一致
- 涉及多个模块、边界复杂或回滚成本较高
- 需要把设计交给低上下文实现者继续执行

如果只是短平快的设计对齐，通常停在确认过的设计结论即可，不必为了形式硬写完整文档。

## 门禁说明

- 仍有关键设计不确定性时，不要直接进入实现
- spec 还未确认时，不要开始实施计划
- 如果用户明确要求 “spec 和实施计划都出完再写代码”，则必须两份文档都审阅并确认后再离开本 skill

## 大任务的处理方式

当请求明显过大时：

1. 先指出当前范围为什么不适合塞进一个 spec
2. 拆出 2-4 个可独立理解的子项目
3. 说明依赖关系和建议顺序
4. 只继续推进第一个子项目

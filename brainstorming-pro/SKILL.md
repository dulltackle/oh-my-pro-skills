# 把模糊想法收敛成可执行设计

本 skill 用于在编码前完成设计对齐。重点是先看现有上下文，判断当前到底需要澄清、比较方案、写设计文档，还是只做范围拆分；不要把所有请求都机械推进到完整闭环。

## 核心定位

- 先理解项目现状，再判断这是不是一个需要设计对齐的问题
- 用逐轮澄清和方案比较，把模糊需求收敛成明确决策
- 只在风险、规模或用户要求足够明确时，才继续产出 spec 与实施计划
- 如果请求已经是明确实现问题，不要强行套用本 skill

<HARD-GATE>
当请求仍存在产品、交互、页面结构或架构层面的关键不确定性时：

- 不要直接写代码
- 不要调用实现型 skill
- 不要脚手架、生成实现文件或开始编码

只有在设计风险已经收敛，且本次任务要求的设计产物已经完成并通过确认后，才允许离开本 skill 进入实现阶段。

如果用户明确要求先出 spec / plan，再进入编码，那么 spec 与实施计划都必须完成审阅并得到用户确认后，才允许进入实现。

如果请求本身只是 bug 修复、明确的小改动、局部重构或已有详细指令，则不要启用这套门禁。
</HARD-GATE>

## 什么时候用

- 目标、边界或成功标准还不清楚，需要先做设计对齐
- 一个功能存在多种合理做法，需要比较方案和取舍
- 页面、流程、组件行为或系统结构需要先统一方向
- 用户明确要求先做设计说明、spec、实施计划，再进入编码
- 当前需求过大，先拆成更小的设计子问题更安全

## 什么时候不要用

- 修 bug
- 小范围重构
- 已明确的低风险配置修改
- 定向文案、测试、文档、依赖或元数据调整
- 需求已经足够清晰，只差落地实现

## 通用判断框架

优先沿着这四个维度判断，而不是立刻进入完整流程：

1. 当前缺的是目标澄清、方案比较，还是文档落地
2. 设计风险是否高到需要书面 spec 和实施计划
3. 这次任务涉及产品逻辑、页面交互，还是系统结构
4. 是否需要先拆分范围，再只推进第一个可落地子问题

## 任务分流

### 设计对齐

- 适用：需求仍模糊，需要先澄清目标、约束、范围、成功标准
- 核心产物：确认过的设计结论、关键取舍和推荐方案
- 默认做法：先看上下文，再逐轮追问，再给 2-3 种方案并推荐
- 只有在风险较高或用户明确要求时，才升级到 spec / plan

### 文档收敛

- 适用：总体方向已经确定，但还需要正式 spec、实施计划或审阅
- 核心产物：spec、实施计划、审阅结论、用户确认
- 这类请求优先读取 `references/deliverables.md` 与对应 reviewer prompt

### 大任务拆分

- 适用：单个请求同时覆盖多个相对独立的子系统或交付物
- 核心产物：拆分后的子项目边界、依赖关系和建议顺序
- 默认只继续推进第一个适合落地的子问题，不把所有内容挤进一个 spec

## 默认工作方式

- 先查看现有代码、文档、模式和近期上下文，再判断是否真的需要本 skill
- 每次只推进一个澄清问题，优先补目标、约束、边界和成功标准
- 默认提供 2-3 种可行方案，说明收益、风险、复杂度和适用条件，再给推荐
- 如果某一段仍有争议，先回到澄清或改方案，不要硬推进写文档
- spec 与实施计划是高风险任务或显式需求的交付物，不是所有请求的默认终点

## 资源入口

- `references/design-workflow.md`：分流后的默认推进方式、提问节奏和门禁说明
- `references/deliverables.md`：spec / 实施计划的写法、路径约定和审阅入口
- `references/visual-companion.md`：页面、交互、信息架构类任务的补充检查清单
- `references/spec-document-reviewer-prompt.md`：spec 审阅 prompt 模板

## 完成条件

以下条件按本次路由成立，不要求每次都走完整闭环：

- 如果本次只是设计对齐：关键方向、边界和推荐方案已经被用户确认
- 如果本次要求 spec：spec 已写成文档，完成必要审阅，并得到用户确认
- 如果本次要求实施计划：实施计划已写成文档，完成必要审阅，并得到用户确认

在以上条件满足之前，保持在“设计与规划”阶段，不要提前进入实现。

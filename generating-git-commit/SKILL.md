# 生成 Git 提交

## 说明

### 步骤 1：安全检查

在创建任何提交之前，请验证：

1. 用户明确请求了提交（不要主动提交）
2. 未请求破坏性操作（强制推送、硬重置、配置更改）
3. 未请求绕过钩子（--no-verify、--no-gpg-sign），除非明确要求
4. 如果请求修改提交，请验证所有条件：
   - 用户明确请求 或 预提交钩子自动修改了文件
   - HEAD 提交是在当前对话中创建的
   - 提交未推送到远程
   - 关键：如果提交失败/被钩子拒绝，绝不修改

### 步骤 2：收集信息

并行运行以下命令：

- `git status --short` - 查看工作区、未跟踪文件和暂存区
- `git diff --cached --name-only` - 查看已暂存文件列表
- `git diff --name-only` - 查看未暂存文件列表
- `git log --oneline -10` - 审查提交消息风格

然后按以下规则确定分析范围：

1. 如果同时存在 `changes`（未暂存更改）和 `staged changes`（已暂存更改），只分析已暂存文件：
   - 使用 `git diff --cached` 查看改动内容
   - 忽略未暂存 diff，不要把未暂存文件纳入本次提交
2. 如果只有 `staged changes`，使用 `git diff --cached` 分析本次提交内容
3. 如果没有任何已暂存文件（所有文件均为未跟踪或未暂存），将全部未跟踪和未暂存文件纳入本次提交：分析所有文件改动内容，在执行时暂存全部文件

### 步骤 3：分析更改并起草消息

1. **先确定提交边界**：
   - 如果仓库同时有已暂存和未暂存更改，本次提交边界固定为"当前已暂存文件"
   - 不要根据未暂存内容改写提交消息，也不要把未暂存文件加入候选列表
   - 如果仓库没有任何已暂存文件，本次提交边界为"所有未跟踪和未暂存文件"，分析全部改动并全部纳入提交
2. **确定提交类型**：参考下方[类型列表](#提交类型)
3. **检查敏感文件**：.env、credentials.json、secrets - 如果存在则警告
4. **起草消息**，遵循格式：

   ```
   <type>: <description>

   [body]

   [optional footer(s)]
   ```

#### 提交类型

| 类型       | 说明                          | SemVer 影响 |
| ---------- | ----------------------------- | ----------- |
| `feat`     | 新功能                        | MINOR       |
| `fix`      | Bug 修复                      | PATCH       |
| `docs`     | 文档更改                      | -           |
| `style`    | 代码格式（不影响逻辑）        | -           |
| `refactor` | 重构（既非功能也非修复）      | -           |
| `perf`     | 性能优化                      | PATCH       |
| `test`     | 添加/修改测试                 | -           |
| `build`    | 构建系统或依赖变更            | -           |
| `ci`       | CI 配置变更                   | -           |
| `chore`    | 其他不修改 src 或 test 的更改 | -           |
| `revert`   | 回滚之前的提交                | -           |

#### 消息规则

- **描述**：使用祈使语气，关注"为什么"，简洁明了（50 字符内最佳）
- **破坏性变更**：
  - 在类型/范围后加 `!`，如 `feat!:`
  - 或在 footer 中添加 `BREAKING CHANGE: <description>`
- **Body**：空一行后添加更多上下文，简短描述当前修改的内容
- **Footer**：可选，如 `Refs: #123`、`Reviewed-by: Z`

### 步骤 4：单次用户确认

整个 git 提交流程只能向用户确认一次。这一次确认既用于确认提交消息，也用于确认提交范围和授权执行；确认完成后，不得因为"用户改了消息"再发起第二次确认。

在执行提交前，必须先以文本输出完整提交信息，再发起确认，**强制等待用户响应**后才能继续；未收到响应前，不得执行任何 `git` 操作。

#### 第一步：无条件输出提交详情（所有环境均执行）

以下内容必须一次性以文本形式呈现：

1. 当前起草的提交消息（完整格式，必须原样展示，包含 subject、body、footer）
2. 将要提交的文件列表
3. 本次提交边界说明（例如"仅提交当前已暂存文件"）
4. 如有敏感文件，显示警告

**推荐呈现结构**：

```text
feat: add user profile endpoint for mobile integration

Add profile read endpoint for mobile clients.

Refs: #123

提交边界：
- 仅提交当前已暂存文件

将要提交的文件：
- src/api/user.py
- tests/test_user.py

警告：
- 未发现敏感文件
```

#### 第二步：发起确认并停止执行（所有环境均执行）

输出提交详情后，立即发起确认并停止，等待用户响应。

**优先**调用当前环境中可以暂停执行、强制等待用户点击选项的结构化交互工具发起确认，呈现以下选项：

- 确认提交（使用上方起草的消息）
- 取消提交（终止流程）
- 提示用户也可以直接输入自定义消息作为回复

如果没有此类工具或调用失败，**再**以格式化文本列出选项后停止执行，等待用户回复：

```
请选择：
  [1] 确认提交（使用上方起草的消息）
  [2] 取消提交
  [其他] 直接输入修改后的提交消息（将视为最终消息直接执行）
```

无论使用哪种方式，均需在同一次输出中明确告知用户：**如果直接回复一条新消息，该消息将作为最终提交消息直接执行，不再二次确认**。

#### 处理响应

- **选"确认提交"**：继续执行步骤 5，使用起草消息
- **选"取消提交"**：终止流程，不执行任何 git 操作
- **其他文本回复**（既非"确认"也非"取消"）：将该文本整体视为用户提供的最终提交消息，直接继续执行步骤 5，不再发起第二次确认

**重要**：

- 未完成这唯一一次确认，绝不执行 `git add` 或 `git commit`
- 如果仓库同时存在已暂存和未暂存更改，确认内容中必须明确说明"仅提交当前已暂存文件"
- 不要因为用户修改了提交消息、补充了 body 或调整了 footer 而再次确认；这些都属于同一次确认内的最终输入
- 只有当用户给出的替代消息明显不完整、含义冲突或不是最终消息时，才暂停执行并回到普通对话澄清；澄清后仍只允许再发起一次新的确认，而不是在旧确认基础上追加第二轮确认

### 步骤 5：执行提交

用户确认后，使用单个顺序 bash 调用：

1. 如果用户选择"确认提交"，使用起草消息执行：`git commit -m "message"`
2. 如果用户选择"修改后提交"，使用用户在该次确认中提供的最终消息执行：`git commit -m "message"`
3. 如果本次提交边界来自未暂存文件，先暂存所有相关文件：`git add <files>`（当没有任何已暂存文件时，暂存全部未跟踪和未暂存文件），再执行 `git commit -m "message"`
4. 验证：`git status`

执行前再次检查：

- 如果仓库同时存在已暂存和未暂存更改，不要运行新的 `git add`
- 不要改变用户已有的暂存集合；直接基于当前暂存区提交
- 进入执行阶段后，不要再向用户确认一次提交消息或文件列表

### 步骤 6：提交后

- 确认提交成功
- 除非明确请求，否则不要推送
- 报告已提交的内容

## 示例

示例 1：新功能
用户："提交这些更改"
分析：为用户服务添加了新的 API 端点
消息：`feat: add user profile endpoint for mobile integration`
暂存文件：src/api/user.py, tests/test_user.py
确认：先以文本输出完整消息、提交边界和文件列表；再通过结构化交互工具（若可用）或格式化文本选项发起确认，等待用户响应后执行

示例 2：Bug 修复
用户："为登录修复创建提交"
分析：跨 3 个文件修复了身份验证错误
消息：`fix: resolve session validation edge case during token refresh`
确认：先以文本输出完整消息；再通过结构化交互工具（若可用）或格式化文本选项等待用户响应，用户接受或回复新消息后，直接提交

示例 3：没有任何已暂存文件
用户："提交一下"
仓库状态：所有文件均为未跟踪或未暂存，无已暂存文件
分析：分析全部未跟踪和未暂存文件的改动内容
消息：`feat: init project with core modules`
确认：先以文本输出完整消息，说明"仓库无已暂存文件，将提交全部未跟踪和未暂存文件"；再发起确认，等待用户响应
执行：`git add` 全部文件后 `git commit`

示例 4：同时存在 staged 和 unstaged changes
用户："帮我把已经暂存的改动提交掉"
仓库状态：`src/auth.ts` 已暂存，`README.md` 仍未暂存
分析：只读取 `git diff --cached`，只根据 `src/auth.ts` 起草消息
消息：`fix: handle token refresh validation failure`
确认：先以文本输出完整消息，明确说明"仅提交当前已暂存文件"；再发起确认，用户一旦确认或回复新消息，直接执行 `git commit`

示例 5：用户回复自定义消息
起草消息：`feat: add new feature`
用户回复：`feat: add real-time analytics widget`
处理：该回复既非"确认"也非"取消"，视为最终提交消息，直接提交，不再二次确认

示例 6：破坏性变更
分析：移除了废弃的 API
消息：

```
feat!: remove deprecated legacy endpoints

BREAKING CHANGE: Legacy API v1 endpoints have been removed. Migrate to v2.
```

示例 7：多段落 body
消息：

```
fix: prevent racing of requests

Introduce request id and reference to latest request.
Dismiss incoming responses other than from latest request.

Refs: #123
```

## 故障排除

错误："没有可提交的内容"
原因：没有更改或所有更改已提交
解决方案：运行 git status 验证，通知用户

错误："预提交钩子失败"
原因：钩子检查失败（代码检查、测试等）
解决方案：绝不修改。修复问题，然后创建新提交

错误："合并冲突"
原因：本地和远程之间存在冲突
解决方案：不要继续。通知用户并请求指导

错误："分离的 HEAD"
原因：不在分支上
解决方案：警告用户，询问他们是否要先创建/切换到分支

情况："用户取消确认"
原因：用户在确认步骤选择取消
解决方案：终止提交流程，不执行任何 git 操作，通知用户已取消

情况："用户回复了一条新的提交消息"
原因：用户希望调整 subject、body 或 footer
解决方案：将该回复整体视为本次提交的最终授权消息，直接执行提交，不再发起第二次确认

情况："同时存在 staged 和 unstaged changes"
原因：工作区中既有已暂存文件，也有未暂存文件
解决方案：仅分析并提交已暂存文件；不要重新 `git add`，也不要把未暂存改动纳入提交

## 反模式

| 避免                          | 改为                                                                   |
| ----------------------------- | ---------------------------------------------------------------------- |
| 未经请求就提交                | 仅在用户要求时提交                                                     |
| 未经确认就提交                | 起草消息后必须获取唯一一次用户确认                                     |
| 自动推送                      | 等待明确的推送请求                                                     |
| 通用消息（"update code"）     | 具体（"fix: null pointer in auth"）                                    |
| 包含机密                      | 警告并排除敏感文件                                                     |
| 修改失败的提交                | 修复后创建新提交                                                       |
| 深入的 git 探索               | 仅运行必要的 git 命令                                                  |
| 混合分析已暂存和未暂存改动    | 同时存在两者时，只分析并提交已暂存文件                                 |
| 无类型的消息                  | 使用 `feat:`、`fix:` 等前缀                                            |
| 过长的描述                    | 保持 50 字符内，详细内容放 body                                        |
| 展示提交消息后直接提交        | 先以文本输出完整信息，再通过结构化交互工具或格式化文本选项等待用户响应 |
| 用户回复新消息后再确认一次    | 将用户的文本回复视为最终授权，直接执行                                 |
| 在 SKILL 中硬编码具体工具名称 | 描述能力特征（"暂停执行并强制等待用户点击选项"），由 agent 自行判断    |

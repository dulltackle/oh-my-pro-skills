# workflow 的使用方式

本文件说明何时应该在 skill 中写 workflow。workflow 是一种可选结构，不是顶层默认范式。

## 先判断是不是该写 workflow

### 开放式任务 skill

适用于需要抽象策略、澄清边界、保留重规划空间的任务。

建议：

- 顶层优先写判断框架，而不是顺序流程
- workflow 如果有价值，只能下沉成某个分支的参考说明
- 不要把“先做 A 再做 B 再做 C”写成默认口径

### workflow 式 skill

适用于顺序稳定、风险明确、对复现要求高的任务。

可以在顶层写顺序流程，但前提是：

- 每一步的输入输出都相对稳定
- 打乱顺序会明显增加失败率
- 模型不需要在现场频繁改写策略

### 混合型 skill

适用于顶层要做判断，但部分子任务必须按固定顺序执行的场景。

推荐做法：

- 顶层只写定位、判断框架和分流
- workflow 只存在于具体分支、reference 或脚本入口附近
- 让调用方先决定“是否进入 workflow”，再看具体步骤

## 顺序流程

适用于必须按顺序执行的子任务。建议先给总览，再给逐步说明。

```md
## 处理流程

1. 理解请求和输入
2. 识别风险与边界
3. 执行核心步骤
4. 验证结果
5. 输出交付物
```

适合：

- 文件转换
- 多步骤工具调用
- 结构化文件生成
- 严格顺序的发布或打包动作

## 条件分支

当 skill 需要根据上下文切换路径时，先给判定条件，再给对应流程。对开放式任务，这通常应该是顶层默认写法。

```md
1. 判断任务类型
   创建新内容 -> 走“创建流程”
   修改旧内容 -> 走“改造流程”

2. 创建流程：...
3. 改造流程：...
```

适合：

- 新建 vs 改造
- 有文件输入 vs 无文件输入
- 自动执行 vs 半自动执行

## 反馈循环

质量敏感任务应内置“执行 -> 校验 -> 修正 -> 再校验”的闭环，但它属于验证分支，不应默认吞掉所有请求。

常见模式：

```md
1. 先完成初稿或产物
2. 立刻运行校验脚本
3. 如果失败，按错误信息修正
4. 直到校验通过再继续
```

适合：

- 结构化文件生成
- schema 校验
- 打包前检查
- benchmark 聚合

## 顶层 `SKILL.md` 的推荐组织

顶层文件建议只保留：

1. 定位与适用场景
2. 设计哲学与边界
3. 通用判断框架
4. 任务入口判断
5. 何时读取哪些 `references/`
6. 何时运行哪些 `scripts/`

如果正文开始出现大段 schema、完整脚本说明、大量示例，或一上来就铺满顺序流程，通常说明应该下沉。

# 写作工作流

在编写或重构 skill 时，优先使用清晰、可分阶段执行的流程，而不是把所有要求堆成大段说明。

## 顺序流程

适用于必须按顺序执行的任务。建议先给总览，再给逐步说明。

```md
## 处理流程

1. 理解请求和输入
2. 识别风险与边界
3. 执行核心步骤
4. 验证结果
5. 输出交付物
```

适合：

- 文档生成
- 代码修改
- 文件转换
- 多步骤工具调用

## 条件分支

当 skill 需要根据上下文切换路径时，先给判定条件，再给对应流程。

```md
1. 判断任务类型
   创建新内容 -> 走“创建流程”
   修改旧内容 -> 走“改造流程”

2. 创建流程：...
3. 改造流程：...
```

适合：

- 新建 vs 改造
- 有文件输入 vs 无文件输入
- 自动执行 vs 半自动执行

## 反馈循环

质量敏感任务应内置“执行 -> 校验 -> 修正 -> 再校验”的闭环。

常见模式：

```md
1. 先完成初稿或产物
2. 立刻运行校验脚本
3. 如果失败，按错误信息修正
4. 直到校验通过再继续
```

适合：

- 结构化文件生成
- schema 校验
- 打包前检查
- benchmark 聚合

## 顶层 `SKILL.md` 的推荐组织

顶层文件建议只保留：

1. 定位与适用场景
2. 任务入口判断
3. 核心原则
4. 通用闭环
5. 何时读取哪些 `references/`
6. 何时运行哪些 `scripts/`

如果正文开始出现大段 schema、完整脚本说明或大量示例，通常说明应该下沉。

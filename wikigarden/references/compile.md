# Compile: 知识编译（Ingest 的核心子操作）

> 目录：[目标](#目标) → [编译产物](#编译产物) → [流程](#流程) → [质量标准](#编译质量标准) → [边界](#边界)

## 目标

将 raw/ 中的原始素材，由 LLM **增量式**地加工整合进 wiki/ 的现有知识体系。

> 这不是"给每篇文章写个摘要"就完了。一次摄入应该让 LLM **真正理解**内容，然后更新 Wiki 中所有受影响的页面——概念页、专题页、索引、关系图。**一个新源可能改动 10-15 个 Wiki 页面。**

## 编译产物

> 以下路径为**逻辑路径**（所有 provider 共用），物理存储由各 provider 自行映射。详见 `providers/INTERFACE.md`。

| 产物 | 逻辑路径 | 说明 |
|------|---------|------|
| 总索引 | `wiki/index` | 知识目录，每页一条目+摘要，Query 的入口 |
| 操作日志 | `wiki/log` | 追加-only 的时间线，可解析格式 |
| 文档摘要 | `wiki/summaries/<slug>` | 每篇素材的结构化摘要 |
| 概念页 | `wiki/concepts/<concept>` | 核心定义、多视角并列、出处标注 |
| 专题页 | `wiki/topics/<topic>` | 跨文档主题综合 |
| 对比页 | `wiki/comparisons/<name>` | 结构化对比分析（从 Query 写回） |
| 关系图 | `wiki/links/relations` | 概念/文档间关联 |
| 术语表 | `wiki/glossary` | 领域术语统一定义 |

## 流程

### 🔒 Step 1: 分析素材 + 制定计划 ⏸️ GATE

> **必须输出编译计划并与用户确认后才能继续。这是整个 Compile 阶段的门控，不得跳过。**

通过 `raw_list(status=pending)` 查询待处理素材，输出编译计划并**与用户确认**（大型知识库尤其重要）：

```markdown
## 编译计划
- 素材总数: N 篇 (X 篇新增 pending)
- 核心主题: [A, B, C]
- 关键概念: [X, Y, Z] — 其中 [X, Y] 已有页面需更新, [Z] 需新建
- 受影响的专题页: [P, Q] — 需追加新素材的发现
- 预计改动: ~M 个 Wiki 页面
```

### 🔒 Step 2: 生成文档摘要

> **每条素材都必须生成摘要，不得省略。**

通过 `raw_read` 读取素材内容，通过 `wiki_write` 写入摘要。

```markdown
# <标题>

## 元信息
- 来源: <url> · 类型: <article/paper/...> · 采集时间: <date>

## 一句话总结
<核心观点，一句话>

## 核心要点
1. ...
2. ...

## 关键概念
- [[概念A]] → wiki/concepts/概念A
- [[概念B]] → wiki/concepts/概念B (需更新：本文补充了 XX 视角)

## 可操作结论
<可直接使用的结论或建议>

## 与其他素材的关系
- 与 <文章1> 的关联: 相互印证/矛盾/互补
- 与 <概念C> 的关联: ...
```

### 🔒 Step 3: 更新概念页（增量，非新建优先）

> **每个识别出的关键概念都必须处理。**

对每个识别出的概念：

- **已有概念页** → 通过 `wiki_read` 读取现有内容，追加新视角/新证据/矛盾标注（不同来源并列，不隐），再通过 `wiki_write` 写回
- **新概念** → 通过 `wiki_write` 创建概念页

#### 概念粒度决策指南

> 从单篇文章中拆出的概念，需要判断是「独立概念」还是「主概念的子特性」。

| 判断维度 | 独立概念（建新页） | 子特性（作为主概念子章节） |
|---------|-------------------|------------------------|
| 跨文档复用性 | 多篇文章都会讨论 | 仅在某篇的特定上下文出现 |
| 抽象层级 | 领域核心术语 | 某个系统的实现细节/设计选择 |
| 独立解释性 | 不依赖原文即可理解 | 必须结合主概念才完整 |

**实践原则：**
1. 单篇文章首次摄入时，子特性优先写入主概念页内的子章节（如 "## 渐进式披露" 作为 Skills 闭环系统页的一节）
2. 当第二篇文章也深入讨论同一子特性时，再考虑拆出独立概念页并交叉引用
3. 避免"单来源浅层"孤岛页面——只有一条出处的概念页长期无法丰富，且让索引臃肿

```markdown
# <概念名>

## 定义
<简洁定义>

## 核心要点
- 要点1 (来源: 文章1)
- 要点2 (来源: 文章1, 文章2)

## 不同视角/争议
<不同素材有不同说法时并列展示>
- 视角A: ... (来源: 文章1)
- 视角B: ... (来源: 文章2)

## 相关概念
- [[A]] · [[B]]

## 出处
- 文章1, 文章2, ...
```

### 🔒 Step 4: 更新专题页

> **必须判断并执行专题页更新，即使结论是「无需更新」也要记录原因。**

跨文档的相关内容整合到对应专题。判断：
- 这个新素材是否改变了专题的结论？
- 是否需要新增子主题？
- 是否发现了之前未注意的联系？

通过 `wiki_read` 读取专题页，评估后通过 `wiki_write` 更新。

### 🔒 Step 5: 维护索引与日志

> **index 和 log 每次 ingest 后必须更新，不得遗漏。**

#### index

通过 `wiki_read` 读取现有 index，追加新页面条目、刷新摘要、更新状态标记，再通过 `wiki_write` 写回。

#### log（canonical 格式定义）

Wiki 中所有操作日志统一使用以下前缀格式，其他 references 文件引用此定义：

> 统一前缀：`## [YYYY-MM-DD] <操作类型> | <标题>`
> 操作类型：ingest | query | lint | enhance | output

通过 `wiki_read` 读取现有 log，**追加**一条记录，再通过 `wiki_write` 写回：

```markdown
## [YYYY-MM-DD] ingest | <文章标题>
- 来源: <url>
- 产出: wiki/summaries/<slug> (新建), wiki/concepts/X (更新), wiki/topics/Y (追加), wiki/index (更新)
- 影响: ~N 个页面
- 备注: <如有值得记录的决策>
```

### 🔒 Step 6: 增量标记

> **必须将已处理素材的 status 改为 processed。**

通过 `raw_update_meta` 将已处理素材的 status 改为 `processed`。只处理 pending 的，不重编整个 Wiki。

## 编译质量标准

### ✅ 好的 Wiki 页面
- 层次清晰，信息有来源标注
- 概念间双向链接
- 提炼后的精华，不是原文复制
- 有"可操作结论"

### ❌ 差的 Wiki 页面（避免）
- 大段复制原文无提炼
- 无来源标注 / 无链接的孤岛
- 两个概念边界模糊讲的是同一件事
- 无来源断言（LLM 幻觉防护）

## 边界

- **增量优先**：只处理新素材，不重编全部
- **矛盾不隐**：不同素材说法不一致时并列展示
- **深度适度**：以"能回答该领域常见问题"为度，不是越详细越好

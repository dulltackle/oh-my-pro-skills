# 发送前你需要确认什么

## 运行前提

- Linux 图形桌面环境
- 已安装并登录桌面版微信，窗口名可被识别为“微信”
- 已安装 `xclip`
- 已安装 `xdotool`

如果缺少 `xclip` 或 `xdotool`，真实发送会在预检阶段失败。

## 默认状态路径

- 默认数据库路径：`./.state/state.db`
- 这个目录由 CLI 自动创建
- 除非用户明确指定 `--db`，否则都使用默认状态路径

## 即时发送场景的最小信息集

- 发文本：联系人 + 文本内容
- 发附件：联系人 + 本地附件路径
- 可选：`--search-index` 指定微信搜索结果位次

如果未传 `--search-index`，必须提前用 `contact --set <联系人> --search-index <N>` 保存联系人搜索结果位次。

## 真实发送边界

- 真实发送必须来自用户明确要求。
- 联系人、文本内容、附件路径缺失或有歧义时必须先追问。
- 不要猜测联系人、附件路径或消息正文。
- 不得改写、润色、补全用户要发送的消息内容。
- `--force` 只能在用户明确要求强制发送或绕过限频时使用。
- 不要把真实发送自动降级成 `--dry-run`。

## `search-index` 语义

`--search-index <N>` 表示微信搜索后用于选择目标结果的 1-based 搜索结果位次。第 1 个结果执行 0 次 `Down`，第 N 个结果执行 N-1 次 `Down`。例如 `--search-index 2` 表示用户希望选择第 2 个搜索结果，CLI 会在搜索联系人后执行 1 次 `Down`。

这个值依赖当前微信搜索结果排序。首次给联系人发送前，先让用户明确目标联系人对应的搜索结果位次，并用 `contact --set <联系人> --search-index <N>` 保存。

## 真实发送 vs 演练发送

| 维度 | 真实发送 | `--dry-run` |
| --- | --- | --- |
| 限频判断 | 会执行 | 不会执行 |
| 依赖检查 | 会执行 | 不会执行 |
| 微信窗口检查 | 会执行 | 不会执行 |
| 真实 GUI 操作 | 会执行 | 不会执行 |
| 发送历史记录 | 会写入 | 不会写入 |

结论：

- `--dry-run` 只用于流程演练。
- `--dry-run` 不能证明当前机器真实可发送。
- 不要把真实发送自动降级成 `--dry-run`。

## 什么时候不要用 `dry-run` 判断结果

以下问题不能靠 `--dry-run` 验证：

- 是否缺少 `xclip` 或 `xdotool`
- 是否能找到并激活微信窗口
- 是否会被静默时段、小时上限、失败冷却或最小发送间隔拦截
- 是否会在真实粘贴、按键或发送动作中失败

## 用户意图到命令的映射

- “发微信给张三：今晚 8 点到”
  - 执行：`send-text --contact 张三 --message "今晚 8 点到"`
- “给李四发这张图 `/tmp/a.png`”
  - 执行：`send-file --contact 李四 --file /tmp/a.png`
- “先演练给王五发一句明天见”
  - 执行：`send-text --contact 王五 --message "明天见" --dry-run`

## 调用前建议检查

- 首次给联系人发送前，先用 `contact --set ... --search-index ...` 保存搜索结果位次。
- 排查发送失败前，先看 `history` 的最近记录。
- 排查限频前，再看 `status --json` 和 `config --json`。
- 用户指定附件时，CLI 会校验文件是否存在并转成绝对路径。

## CLI 自身负责的机制

这个 skill 只调用 CLI，不重新实现发送机制。以下逻辑由 CLI 自己处理：

- SQLite 状态存储
- 基础限频判断
- 联系人搜索位次
- 文本/附件发送
- 失败记录与失败摘要
